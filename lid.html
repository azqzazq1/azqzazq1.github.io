LID | Linux Integrity Drift

Systematic discovery of kernel code paths that bypass LSM security hooks
5 findings LSM bypass kernel 5.2+ AppArmor SELinux io_uring eBPF mount API AF_XDP

The Core Idea

The Linux Security Module framework has maintained a single guarantee for over 20 years: security modules can only add restrictions, never remove them. This guarantee is correct. LID does not break it.

LID finds something different. It identifies kernel subsystems that perform security-sensitive operations without consulting the LSM framework at all. The gate was never breached. It was walked around.

The security policy is enforced correctly. The problem is that the kernel has code paths that never ask.

LSM Hook Architecture

The kernel uses the call_int_hook macro to iterate through registered security modules. Each module returns 0 (allow) or nonzero (deny). The loop short-circuits on the first denial, ensuring modules can only add restrictions.

Detailed Findings

LID-001 | Critical Visibility Gap

eBPF Pathname Rewriting

A BPF kprobe attached to do_sys_openat2 uses bpf_probe_write_user to rewrite the filename in the user space buffer before the kernel copies it. Combined with a hard link at an AppArmor-allowed path, the kernel resolves the rewritten path, AppArmor evaluates the allowed path instead of the denied one, and the access succeeds with zero audit trace.

The attack exploits a fundamental gap between AppArmor's pathname-based access control model and eBPF's ability to operate before any LSM hook fires. Two hard links to the same inode are treated as entirely different files by AppArmor policy. The BPF program selectively targets specific process and path combinations, making the bypass transparent, selective, and audit-invisible.

LID-002 | Critical Visibility Gap + Practical Escalation

io_uring MSG_RING File Descriptor Transfer

The io_uring MSG_RING operation allows one io_uring instance to inject a file descriptor into another ring's fixed file table. This fd transfer completely bypasses security_file_receive(), the LSM hook that AppArmor and SELinux rely on for cross-process file descriptor validation.

Unlike other LID findings, this vector operates from unprivileged userspace with no special capabilities required. A regular user can transfer arbitrary file descriptors between io_uring instances without any LSM evaluation. The transferred fd is invisible to every registered security module.

LID-003 | High Visibility Gap

New Mount API Bypass

The new mount API introduced with fsopen(), fsmount(), and move_mount() bypasses security_sb_mount(), which is the primary hook that AppArmor and SELinux use to enforce mount policy. The new API path performs mount operations through a different code path that was never wired into the existing LSM mount hook.

LID-004 | High Visibility Gap

BPF Token AppArmor Blindness

The BPF token subsystem (kernel 6.9+) introduces nine security hooks for token creation, delegation, and usage. AppArmor implements zero of these nine hooks. Any BPF token operation occurs with complete AppArmor blindness, no policy evaluation, and no audit logging.

LID-005 | Medium Visibility Gap

AF_XDP tc Egress Bypass

Raw network frames transmitted through AF_XDP sockets bypass the tc (traffic control) egress path entirely. Container environments that rely on tc filters for network policy enforcement (including some configurations of Docker and plain iptables setups) are blind to AF_XDP transmitted traffic. The frames go directly to the NIC driver queue, skipping all tc qdisc and filter processing.

Architectural Implications

LID reveals a design tension in the Linux security architecture. The LSM framework correctly enforces that modules can only add restrictions. The eBPF subsystem provides unrestricted hook points that operate before LSM checks. Newer kernel subsystems (io_uring, the new mount API, BPF tokens) introduce operations that were never connected to existing LSM hooks.

The result is that the security perimeter, as understood by AppArmor and SELinux policy, does not match the actual kernel attack surface. Defenders who rely on LSM-based policy enforcement and audit logging have blind spots in their visibility model. Operations can occur, data can be accessed, and file descriptors can be transferred without any security module being consulted.

This is not a bug in any individual component. It is the natural consequence of a kernel that evolves faster than its security framework.
